NEPRI expliciet toestaan en meegeven als secundaire identifier voor onderwijsmedewerkers in de Employees API.

Beschrijving

Wij willen dat in de Employees API bij het object Employee expliciet wordt vastgelegd dat nlEduPersonRealId (NEPRI) als secundaire identifier voor onderwijsmedewerkers gebruikt mag worden en via userIds wordt meegegeven wanneer deze bij de bron bekend is.

Op dit moment kan een onderwijsmedewerker via Entree wel een koppelbare identiteit hebben, namelijk de NEPRI, terwijl dezelfde medewerker via de Edu-V Employees API alleen een ASI/LAS-key ontvangt. Voor afnemers is deze ASI niet voldoende om de onderwijsmedewerker betrouwbaar te koppelen aan de bestaande auth-user uit Entree. Daarnaast lijkt er geen reden dat NEPRI niet voor medewerkers beschikbaar zou moeten zijn, Magister stuurt dit netjes mee in Entree maar kan volgens de specs dit niet meesturen in Edu-V. Hierbij ontstaat een situatie dat ik gebruikers niet kan koppelen als deze geen ECK_ID of NEPPI hebben.

De gewenste wijziging is daarom:

  1. Leg expliciet vast dat NEPRI een toegestane secundaire identifier is voor onderwijsmedewerkers binnen Employee.userIds.

  2. Leg vast dat een bron deze identifier moet meegeven wanneer deze beschikbaar is.

  3. Zorg dat de Employees API-specificatie en de algemene pagina over digitale identiteiten hierin consistent zijn.

Onderbouwing van nut of noodzaak

Afnemers die gebruikers uit Edu-V moeten koppelen aan authenticatiegebruikers uit Entree hebben minimaal één stabiele, ketenbreed herkenbare identifier nodig. Bij onderwijsmedewerkers is niet altijd een ECK iD beschikbaar of gekoppeld. Wanneer in dat geval alleen een ASI/LAS-key wordt geleverd, kan de afnemer de medewerker niet betrouwbaar koppelen aan de bestaande auth-user uit Entree.

Door NEPRI expliciet toe te staan en mee te geven voor onderwijsmedewerkers wanneer deze beschikbaar is, ontstaat een duidelijke en uitvoerbare fallback voor koppeling, zodat ik nooit met alleen een ASI zit.

De wijziging beperkt zich tot het verduidelijken en toepassen van een bestaande identifier voor onderwijsmedewerkers. Er hoeven geen extra persoonsgegevens te worden geïntroduceerd; het gaat om het beschikbaar maken van een identifier die nodig is voor betrouwbare koppeling tussen Edu-V-data en authenticatie via Entree.
